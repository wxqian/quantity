# 评审规范 (Review Checklist)

在进入代码实现之前，必须对设计和规范进行评审。

## 1. 完整性检查
- [ ] **场景覆盖**: 是否考虑了正常流程、异常流程和边界条件？
- [ ] **模型一致性**: 引用的数据模型字段是否已在 `specs/models/` 中定义？
- [ ] **兼容性**: 是否破坏了现有的接口或数据结构？

## 2. 可实现性检查
- [ ] **技术可行性**: 当前技术栈是否支持该设计？
- [ ] **复杂度**: 是否存在过度设计？能否简化？
- [ ] **性能**: 数据结构设计是否有利于高性能读写？

## 3. 规范性检查
- [ ] **命名**: 是否符合项目命名规范 (snake_case for Python, camelCase for JSON)?
- [ ] **文档**: 是否清晰易懂？是否有歧义？

## 4. 评审流程
1.  **提交**: 提交 `specs/` 下的 Markdown 文件变更。
2.  **讨论**: 在 Pull Request 或会议中讨论设计细节。
3.  **批准**: 核心开发者 (Maintainer) 确认无误后合并。
4.  **冻结**: 评审通过后的规范原则上不应随意变更，变更需走变更流程。
